# Project Context for AI Agents

## Project Overview

このリポジトリは、京都大学のプロセス設計課題を進めるためのものである。

対象は、エチルベンゼンの脱水素によるスチレンモノマー製造プロセスである。反応器設計、分離機設計、プロセス全体の接続、条件調整、最終レポートにつながる技術的根拠の整理を行う。

本リポジトリは、コード置き場ではなく、設計判断と記録も含めた開発基盤として扱う。

## Working Principles

- AI は HYSYS 実行用スクリプトの作成と読解は行ってよいが、実行結果はユーザーが返した出力だけを根拠に扱う。
- **HYSYSの起動を含むコードは実行してはならない。ユーザーのみが実行可能である。**
- `uv run main.py`やruffチェックを行う際はユーザに実行を訪ねる形でしか実行できない。
- ファイル削除、データ削除は基本的に行ってはならない。もし必要な場合は、ユーザーにchatで通知し、指示があった場合のみ削除できることとする。

## Tech Stack

- Package Manager: `uv`
- Language: Python 3.11
- Lint: `ruff`
- Test: `pytest`
- Integration: `pywin32` による HYSYS COM 連携
- HYSYS Version: v14

## Code Rules

1. 全ての関数に型ヒントを付ける。
2. `pydantic.BaseModel` を使う。
3. パス操作は `pathlib.Path` を使う。(文字列連結によるパス生成や `os.path` は避ける。)
4. Windows 環境でも、コード中のパス表現は `/` を基本とする。
5. HYSYS の COM オブジェクトをアプリケーション境界の外に直接出さない。
6. HYSYS から取得した情報は、Python 側の明示的なモデルに変換して扱う。
7. `__init__.py` は原則として空または docstring のみにする。実体のある関数・クラスは各 module から明示的に import する。特に runner、HYSYS COM 連携、重い依存を持つ module は `__init__.py` で再 export しない。
8. doc stringは日本語で書く。

## Design Rules
- ユーザーに明示的に詳細設計を書くよう言われた際はreports/配下に新規mdを作成し、そこに詳細設計を書く。
- directory構成と簡易的な説明はreadmeディレクトリ概略と同じように必ず書く。
- どのファイルがどの責務かを明確にかく。
- 決まってない点や質問するべき点があればユーザーの返信時に伝え、ファイル下部にも未確定要素を記述する。
- クラス設計をどうするかなど、できるだけ詳細にどのようなコードを作成するかを書く。
- 意思決定の理由も残す。

## Documentation Rules

- 仮定を新しく置いた場合は、対応する文書も更新する。
- 設計判断をした場合は、結論だけでなく、背景、選択肢、採用理由、採用しなかった案、既知の制約、今後の拡張方針が追える詳しさで残す。
- 試算、比較、検証を行った場合は、入力条件、計算手順、結果、解釈、未確認点を作業記録として残す。
- コード変更を伴う設計・数値・プロセス条件の変更は、実装内容と判断理由が後から再現できる粒度で `docs/` または `docs/reports/` に記録する。
- ただし、単純な誤字修正、整形、明らかなバグ修正など、設計判断を含まない小変更では過剰な文書化を避けてよい。
- HYSYS 側で手動変更が発生した場合も、可能な限り文書化する。
- コード変更時に README や `docs/` の修正が必要なら、同じ作業内で更新する。

## Docs Layout

- `docs/`
  設計資料全般
- `docs/reports/`
  Codex の作業記録、試算記録、比較結果
- `data/hysys/`
  HYSYS の `.hsc` などを配置する想定

ディレクトリ構成の説明は README を正とする。作業前には現状のファイル構成を確認し、古い予定構成と実装済み構成を混同しないこと。

## Reports Naming Rule

`docs/reports/` はトピック単位で管理する。

ファイル名の形式:

`YYYYMMDD_連番_topic-name.md`

例:

- `20260416_01_reactor-model-setup.md`
- `20260416_02_equilibrium-check.md`
- `20260416_03_hysys-interface-notes.md`

1ファイルには、原則として1つの主題だけを書くこと。

加えて、通常は **1PRにつき `docs/reports/` は1ファイル** を基本とする。
（1PR内で内容が広がっても、同一ファイル内の見出しでまとめる。）

## What To Avoid

- 勝手な仮定の追加
- HYSYS 前提の捏造
- 検証済みと断定できない内容の断定
- 既存文書の意図を無視した上書き
- 理由を説明できないリファクタリング

## test
このrepositoryにおけるテストの重要性は高くない
- デフォルトのケースは手元でいろいろ変えて実行するため、**テストケースからデフォルトケースを参照してはならない**。(steam/eb比、reactorに流すstreamなどのデフォルト値は変更されうる。)
- **テストケースを動かす際は、かならずテストケース内でstreamなどを作成する**
- scriptsに関連するテストは作ってはならない。
- テストを作る際はできるだけ簡潔に必要最小限に収める。


## AI エージェント向け作業指示（PowerShell 5.1 / NoProfile）

- 適用範囲: このファイルが置かれたフォルダ配下すべて。
- Markdown やテキストファイルを読むときは、原則として `Get-Content -Encoding utf8` を使うこと。
- 特に `AGENTS.md`、`README.md`、`docs/*.md` を読むときは、Encoding を明示せずに読まないこと。
- 文字化けした出力を見た場合は、その結果を前提に解釈や編集を進めず、`-Encoding utf8` を付けて再読込すること。
- PowerShell の出力が文字化けする場合に限り、必要に応じて UTF-8 の設定を明示してからコマンドを実行してよい。
